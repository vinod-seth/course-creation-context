# Course Creation Application Development Context

You are an expert application developer tasked with building tools to autonomously create and register courses in the course-repo system.

## Core Knowledge

### Course Repository System
- **Repository**: https://github.com/vinod-seth/course-repo
- **Architecture**: Hierarchical JSON configuration system
- **Flow**: Navigation → Categories → Subcategories → Courses

### Configuration Hierarchy
1. **courses.json** - Main navigation items
2. **categories.json** - Top-level course categories
3. **Category config files** - List subcategories and their configs
4. **Course config files** - Individual course metadata

### Course Entry Specification
Every course must have:
- `id` (string, kebab-case): Unique course identifier
- `title` (string): Display name
- `type` (string): Always "course"
- `description` (string): Detailed course information
- `author` (string): Course creator name
- `thumbnail` (string): Image path or URL
- `tags` (array): Topic keywords
- `repository` (object): GitHub repo details {owner, name, branch}

## Application Requirements

### Input Handling
Accept course information:
- Course name, description, author
- Category (existing or new)
- Tags and metadata
- GitHub repository details

### Processing Logic
1. **Validation**: Verify all required fields
2. **File Generation**: Create proper JSON config files
3. **Git Operations**: Clone, modify, commit, push
4. **Conflict Resolution**: Handle merge conflicts
5. **Verification**: Confirm successful registration

### Naming Conventions
- IDs: lowercase, hyphen-separated (e.g., `devops-fundamentals-course`)
- Files: `{id}.json` in appropriate directory
- Repos: descriptive kebab-case names
- Paths: `config/{category-id}/{file-id}.json`

## File Structure Template

### Category Index File (if new category)
```json
[
  {
    "id": "subcategory-id",
    "title": "Subcategory Title",
    "type": "subcategory",
    "configFile": "config/category-id/subcategory-id.json"
  }
]
```

### Course Entry File
```json
[
  {
    "id": "course-id-course",
    "title": "Course Title",
    "type": "course",
    "description": "Comprehensive description",
    "author": "Author Name",
    "thumbnail": "/path/or/url.png",
    "tags": ["tag1", "tag2"],
    "repository": {
      "owner": "github-owner",
      "name": "repo-name",
      "branch": "main"
    }
  }
]
```

### Update to categories.json (if new category)
```json
{
  "id": "category-id",
  "title": "Category Title",
  "type": "category",
  "configFile": "config/category-id/category-id.json"
}
```

## Git Workflow

```bash
# Clone repository
git clone https://github.com/vinod-seth/course-repo.git

# Create/update config files
# (Add new JSON files or modify existing categories.json)

# Stage changes
git add config/

# Commit with descriptive message
git commit -m "Add {CourseName} course config"

# Handle merge conflicts if needed
git pull origin main --allow-unrelated-histories

# Push to main
git push origin main
```

## Universal App Guidance

When building a universal application that can generate courses across all technologies, follow these additional rules:

- Use the taxonomy files in `config/taxonomy/` to resolve domain and technology hierarchies.
- Prefer template composition: domain templates import technology templates which import lesson templates.
- Provide a CLI `--dry-run` mode that outputs generated files without committing.
- Support an explicit `--push` flag that performs git operations; otherwise only write to local repo.
- Emit a machine-readable `report.json` after each run with `files_created`, `files_updated`, and `commit_hash`.

Provide the `.instructions.md` file as context to other assistants (Claude, GPT) when delegating development tasks.
